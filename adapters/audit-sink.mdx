Every call to `RuntimeEngine.runCommand` produces a single `AuditRecord`, regardless of whether the command succeeds, is denied by a guard or policy, hits a constraint, or throws an unexpected error. You receive that record by providing an `AuditSink` in `RuntimeOptions`.

## What gets recorded

The runtime emits exactly one record per `runCommand` invocation. The record is produced in a `finally` block, so even unhandled errors produce an audit entry before propagating.

The `AuditRecord` shape, as defined in `@angriff36/manifest/audit`:

```typescript
interface AuditRecord {
  /** Stable record id. Generated via RuntimeOptions.generateId (falls back to crypto.randomUUID). */
  recordId?: string;
  /** Wall-clock timestamp ‚Äî milliseconds since epoch. From RuntimeOptions.now. */
  occurredAt: number;
  /** Tenant the command ran against, when known. */
  tenantId?: string;
  /** Organization the actor belonged to (e.g. Clerk orgId). */
  orgId?: string;
  /** Acting user id. */
  actorId?: string;
  /** Caller-supplied request id for trace correlation. */
  requestId?: string;
  /** Origin surface ('route', 'job', 'cli', ...). */
  source?: string;
  /** Entity name the command targeted, when entity-scoped. */
  entity?: string;
  /** Command name. */
  command: string;
  /** Canonical "<entity>.<command>" identifier, when available. */
  commandId?: string;
  /** Outcome classification. */
  outcome: CommandOutcome;
  /** Structured runtime diagnostics, if any. */
  diagnostics?: unknown;
  /** Names of events emitted by the command, when the outcome was success. */
  emittedEventNames?: string[];
  /** IR provenance hash at execution time, for drift detection. */
  irHash?: string;
}
```

### Outcome values

The `outcome` field is one of:

| Value | Meaning |
|-------|---------|
| `success` | Command completed; state was mutated and events emitted. |
| `guard_denied` | A guard expression evaluated to false. |
| `policy_denied` | A policy check rejected the command. |
| `constraint_failed` | One or more constraint validations failed. |
| `concurrency_conflict` | Optimistic concurrency check failed. |
| `missing_tenant_context` | The command required a tenant context that was not supplied. |
| `error` | An unexpected error occurred during evaluation. |

<Note>
`diagnostics` carries the structured failure details ‚Äî `policyDenial`, `guardFailure`, `concurrencyConflict`, `constraintOutcomes`, or `error` ‚Äî so downstream tooling can reason about a denial without reshaping the runtime contract.
</Note>

## First-party sinks

Two sinks ship with the package:

<CodeGroup>

```typescript MemoryAuditSink (development)
import { MemoryAuditSink } from '@angriff36/manifest/audit/memory';

const sink = new MemoryAuditSink();

// Retrieve records after commands run:
const records = sink.getRecords();
```

```typescript PostgresAuditSink (production)
import { PostgresAuditSink } from '@angriff36/manifest/audit/postgres';
import { Pool } from 'pg';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const sink = new PostgresAuditSink(pool);
```

</CodeGroup>

`MemoryAuditSink` is idempotent by `recordId` ‚Äî re-emitting a record with the same `recordId` does not create a duplicate. Use it in tests and sample applications.

`PostgresAuditSink` is backed by `pg` and uses `INSERT ‚Ä¶ ON CONFLICT (record_id) DO NOTHING` for idempotency. Apply the included schema before using it:

<Tip>
The schema file `postgres.sql` is included in the published package. Find it at `node_modules/@angriff36/manifest/src/manifest/audit/sinks/postgres.sql` and run it against your database once during deployment.
</Tip>

## Wiring in an audit sink

Pass your sink via `RuntimeOptions.auditSink`:

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { PostgresAuditSink } from '@angriff36/manifest/audit/postgres';
import { Pool } from 'pg';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

const runtime = new RuntimeEngine(ir, {
  userId: session.userId,
  tenantId: session.tenantId,
  auditSink: new PostgresAuditSink(pool)
});

// Every runtime.runCommand(...) call now writes one AuditRecord.
const result = await runtime.runCommand('createOrder', input, {
  entityName: 'Order',
  instanceId: orderId
});
```

## Implementing a custom sink

You can implement `AuditSink` against any backend ‚Äî OpenTelemetry, a file logger, a remote webhook, or your own table schema:

```typescript
import type { AuditSink, AuditRecord } from '@angriff36/manifest/audit';

export class OpenTelemetryAuditSink implements AuditSink {
  async emit(record: AuditRecord): Promise<void> {
    // Write to your observability backend
    telemetry.track('manifest.command', {
      outcome: record.outcome,
      entity: record.entity,
      command: record.command,
      actorId: record.actorId,
      tenantId: record.tenantId
    });
  }
}
```

Sinks **must** be idempotent against `recordId` when one is supplied, so retries and at-least-once delivery from upstream workers do not double-write.

## Failure policy: fail-open

<Warning>
If `AuditSink.emit` throws, the runtime catches the error, logs it to `stderr`, and **continues normally**. The command result returned to your caller is not affected.

The audit subsystem is observability, not enforcement. Monitor for the log line:

```
[Manifest Runtime] AuditSink.emit failed
```

Alert on it in production so you know when audit records are being lost.
</Warning>
