The `manifest` CLI compiles `.manifest` source files to Intermediate Representation (IR), generates framework-specific code from that IR, and validates the output against the IR schema. You can run each stage independently or chain them together with a single `build` command. A config file lets you store project-level defaults so you rarely need to pass flags by hand.

## Installation

Install the CLI as a development dependency in your project:

```bash
npm install -D @angriff36/manifest
```

If you are working inside the Manifest monorepo, you can run commands through the workspace script instead:

```bash
npm run manifest --help
```

## Two build workflows

### Workflow 1 — interactive setup, then build

Run `manifest init` once to create a `manifest.config.yaml` that captures your project's paths and options. After that, every subsequent build uses those defaults:

```bash
# One-time setup — asks a few questions and writes manifest.config.yaml
manifest init

# Every build after that is just:
manifest build
```

This is the recommended approach for any project you will build repeatedly.

### Workflow 2 — direct commands

Pass all options inline. Useful for one-off runs, CI pipelines, or when you want full control without a config file:

```bash
# Compile a single source file to IR
manifest compile modules/recipe.manifest --output ir/

# Generate Next.js routes from that IR
manifest generate ir/recipe.ir.json --projection nextjs --output app/api/

# Or do both steps at once
manifest build modules/recipe.manifest --projection nextjs --code-output app/api/
```

## Command quick reference

| Command | Description |
|---|---|
| `manifest init` | Interactive setup that creates `manifest.config.yaml` |
| `manifest compile [source]` | Compile `.manifest` source files to IR JSON |
| `manifest generate <ir>` | Generate code from IR using a projection |
| `manifest build [source]` | Compile and generate in one step |
| `manifest validate [ir]` | Validate IR JSON against the IR schema |

## Config file priority

When a command resolves an option it checks three sources in order:

1. **CLI flags** — values you pass directly on the command line (highest priority)
2. **`manifest.config.yaml`** — project-level defaults loaded from the project root
3. **Built-in defaults** — fallback values baked into the CLI (lowest priority)

This means you can override any config value for a single run without editing the file. For example:

```bash
# Normally uses config's code-output path; override just for this run
manifest build --code-output apps/staging/app/api/
```

<Tip>
The CLI also accepts `manifest.config.yml`, `.manifestrc.yaml`, and `.manifestrc.yml` as config file names, checked in that order after `manifest.config.yaml`.
</Tip>
